# TriviumDB v0.4.82 索引性能与精度评测报告

> 测试环境：Windows 11 x86_64 / Rust 1.93.1 (release profile, LTO=true)  
> 测试时间：2026-04-07  
> 测试数据：固定随机种子（`StdRng::seed_from_u64`），聚类倾斜生成的真实分布（模拟人类输入）

## 一、背景与换代说明

TriviumDB 引入了历经重构的 **ERPC v2（Electoral Residual Product Code）** 近似索引，采用 **IVF-FastPQ（倒排+4-Bit 乘积量化）** 架构彻底替代了原有的旧结构（及 HNSW 方案）。

本报告量化评估新的 ERPC 在结构解耦和底层 SIMD 友好度极大提升后的精度（Recall@K）与性能（QPS）表现。

> **核心设计哲学**：ERPC 将 1536 维的高维浮点向量，硬核压缩进了一个仅 64 位的 `u64` 中（最高 16 位为 Electoral ID 倒排桶编号，低 48 位容纳 12 组 4-Bit PQ 残差密码本编码）。配合寄存器级的 Asymmetric Distance Computation (ADC) 查表算法，实现了近乎无限带宽的内存极速扫描。

---

## 二、ERPC vs BruteForce：精度与加速比实测

### 小规模（10,000 条 / 128 维）

极限调优参数下（扫 35% 桶，重排 20%）：

| 策略                  | Recall@10 | QPS       | 加速比     |
| --------------------- | --------- | --------- | ---------- |
| BruteForce（暴力基线） | 100.00%   | 1,811     | 1.00x      |
| BQ 线性组扫 (精查5%)   | 87.50%    | 3,237     | 1.79x      |
| **ERPC (IVF-FastPQ)** | **87.20%**| **3,715** | **2.05x**  |

> 📌 **极限速度探索（Speed 阀门全开）**
> 在此前的测试中，如果我们降低二次检查放行率 (`bq_refined_count`)，同等数据下 ERPC 将以 **67.70%** 的召回精准度，跑到恐怖的 **9,143 QPS** (较 BruteForce **提速 6.0 倍**)。参数由 `effort` 全面接管暴露。

## 三、ERPC vs 传统 HNSW：降维打击

为什么 TriviumDB 坚决不用业内主流的 HNSW 等图结构？

| 指标维度          | 典型图结构 (如 HNSW)                 | **ERPC v2 (IVF-FastPQ)**                       | 选型总结                        |
| ----------------- | ------------------------------------ | ---------------------------------------------- | ------------------------------- |
| **内存与指针**    | 海量堆内存碎片，庞大的 Ghost Nodes | **真正的“零指针”**，数据仅是一维 `u64` 扁平数组 | **ERPC 极轻 🍃**                 |
| **磁盘冷启动**    | 首次启动图重建 / 磁盘反序列化缓慢  | Mmap 下原生直接映射内存地址（`bytemuck` 强转）| **ERPC 毫秒启动 🚀**             |
| **高维 Cache Miss** | 1536维邻居图随机跳跃，L1/L2 命中率灾难 | 物理相邻分布，循环执行 SIMD ADC 全速查表       | ERPC 完美迎合 CPU prefetcher 🧠 |

> 👨‍💻 **深度技术解读**
> 实验表明，在传统高维（1536维）查询中，由于计算一次余弦点积的时间成本极其高昂，传统方法使用图漫游来试图减少计算点积的次数。但是，一次未命中缓存的内存跳转（Cache Miss）通常会直接吃掉几百到上千个时钟周期！  
> 
> ERPC 反其道而行之，使用 **全局 4-Bit 粗密码本**，彻底放弃在第一阶段算浮点余弦，将 1536 维全变为了几个查表指令；这就意味着我们能够以暴力地姿态横推几千个极其便宜的向量而无需内存离散跳转。这就是所谓的**硬件感知型加速引擎**！

## 四、总结与工程美学：曲线平滑度控制

目前 TriviumDB 在建库与查询阶段依靠内部代数曲线控制参数（`n^p / (half_p + n^p)`）：

1. **自动前后台分离**：由于 K-Means 的重度聚类在海量数据下会占用 CPU 长达数秒乃至数分钟，TriviumDB 保持了**只读 Mmap 结合动态 MemTable** 的完美双轨架构。前台插入顺滑无比，满 20,000 条自动在后台异步重建整个 ERPC。
2. **纯真理的取舍 (Trade-Off)**：当对 Agent 用户需要“随带走，快挂载”的数据库时，HNSW 那样臃肿的模型是不可接受的。ERPC 成功验证了通过数学公式将近似召回、磁盘碎片化和内存占用这三大约束死死拿捏住的终极平衡！

# TriviumDB v0.6.0 索引性能与精度评测报告

> 测试环境：Windows 11 x86_64 / Rust 1.93.1 /i5 14400F (release profile, LTO=true)  
> 测试时间：2026-04-28 
> 测试数据：固定随机种子（`StdRng::seed_from_u64(42)`），50 簇聚类倾斜生成的真实分布（模拟句向量语义簇边界漂移）  
> 对比对手：`instant-distance` v0.6（纯 Rust HNSW 实现）

---

## 一、背景与换代说明

TriviumDB 引入了全新的 **BQ（Binary Quantization）三阶段火箭**近似索引，彻底取代了传统的 HNSW 图结构方案。

本报告在从 **5 万 / 20 万 / 50 万条 × 512 ~ 1536 维**的阶梯规模下，量化评估 BQ（不同精查阀门）与 HNSW 和暴力搜索（BruteForce）之间的精度（Recall@K）与性能（QPS）表现。

> **核心设计哲学**：BQ 将 1536 维的高维浮点向量，压缩为仅 1536 个比特位的二进制指纹，通过 Hamming 距离 + `Popcount` 硬件指令做超高速粗排，再取候选集精排 f32 原始向量，整个管线无索引图、无指针跳跃，完美适配 CPU 硬件 Prefetcher 的线性预取机制。

---

## 二、BQ 精查阀门精度 & 性能评测（`bq_vs_brute` 精度测试）

### 2.1 小规模：50,000 条 / 512 维 / Top10 / 100 次查询

| 策略 | Recall@10 | QPS | 加速比 |
| ---- | --------- | --- | ------ |
| BruteForce（基准真值） | 100.00% | 197.8 | 1.00x |
| **BQ（精查 5%）** | **97.00%** | **533.9** | **2.70x** |
| BQ（精查 1%） | 77.80% | 915.8 | 4.63x |

> 📌 **小规模（5 万条）结论**：5% 精查率下召回率依然高达 97%，是高精度场景的最优解。1% 精查率召回率骤降至 77.8%，原因是候选池仅有 500 个名额，聚类噪声下极易漏掉真实最近邻。

---

### 2.2 大规模：200,000 条 / 1536 维 / Top10 / 20 次查询

| 策略 | Recall@10 | QPS | 加速比 |
| ---- | --------- | --- | ------ |
| BruteForce（基准真值） | 100.00% | 7.8 | 1.00x |
| **BQ（精查 5%）** | **99.50%** | **69.8** | **8.96x** |
| BQ（精查 1%） | 89.32% | 174.6 | 22.41x |

> 📌 **大规模（20 万条）结论**：大规模下相同的 1% 精查率候选池扩到了 2000 个，召回率神奇地苟住在 89.32%，同时以 22.4 倍加速比狂飙至 174.6 QPS。中低精度要求的高吞吐场景（如游戏 NPC 联想、模糊商品推荐）可优先考虑此档位。

---

### 2.3 Criterion 精密计时结果（200k × 1536d）

| 策略 | 单次查询时长 | 折算 QPS |
| ---- | ----------- | -------- |
| BruteForce | ~24.5 ms | 40.8 |
| BQ（精查 5%） | ~14.5 ms | 68.9 |
| **BQ（精查 1%）** | **~5.74 ms** | **174.2** |

> Criterion 精密计时排除了 eprintln 打印产生的 I/O 噪声，为最可信的延迟参考值。BQ 1% 单次查询仅需 **5.7 毫秒**，较暴力 **快 4.3x**。

## 三、BruteForce vs BQ vs HNSW：50 万级终极实测（`bench_500k`）

### 3.1 测试参数

| 参数 | 值 |
| ---- | ---- |
| 数据规模 | 500,000 条 |
| 向量维度 | 1536 |
| Top-K | 10 |
| 查询数量 | 20 |
| BQ 精查率 | 5%（`bq_candidate_ratio = 0.05`） |

### 3.2 性能与精度总览

| 策略 | Recall@10 | QPS | 相对 BruteForce 加速比 |
| ---- | --------- | --- | ---------------------- |
| BruteForce（绝对真值基线） | 100.00% | 2.3 | 1.00x |
| **BQ 三阶段火箭（TriviumDB）** | **99.00%** | **26.6** | **11.61x** |
| HNSW（instant-distance） | ~98.00% | 88.0 | 38.26x |

### 3.3 建库耗时对比（这才是关键！）

| 策略 | 建库 / 入库耗时 |
| ---- | --------------- |
| HNSW（instant-distance） | **6,079.04 秒（~1.69 小时）** |
| **TriviumDB** 插入 + flush | **12.16 秒** |

> 📌 **建库速度相差约 500 倍**。AI Agent 记忆数据随时在增删更新，根本不可能接受每次增量后重建数小时的索引。TriviumDB BQ 随写随查，无需重建。

### 3.4 内存消耗对比

| 策略 | 峰值内存占用 |
| ---- | ------------- |
| HNSW | ~6.5 GB（原始向量 × 2 副本 + 多层图边指针） |
| BQ (TriviumDB) | ~3.1 GB（仅原始向量 + 扁平 1-bit 指纹数组） |

## 四、BQ vs HNSW：全维度对比

| 指标维度 | HNSW（图结构） | **BQ 三阶段火箭** | 结论 |
| -------- | -------------- | ----------------- | ---- |
| **建库速度** | 6,079 秒 | 12 秒 | **BQ 快 500x** 🚀 |
| **Recall@10** | ~98.00% | 99.00% | **BQ 更高** ✅ |
| **搜索 QPS** | 88.0 | 26.6 | HNSW 快 3.3x |
| **内存占用** | ~6.5 GB | ~3.1 GB | **BQ 省 50%** 🍃 |
| **冷启动** | 需加载全图（分钟级） | Mmap 直接映射（毫秒级） | **BQ 秒启** ⚡ |
| **删除代价** | Ghost Node + 图边重连 | 墓碑占位，零维护 | **BQ 零负担** |
| **SSD 磨损** | 随机写入（图指针更新） | Append-Only + Compaction | **BQ 友好** 💾 |

> ⚠️ **关于 HNSW 搜索速度领先的澄清**：HNSW 通过"提前绘制精密路线图（1.69 小时）"换来了查询时 3.3 倍的速度优势。但 26.6 QPS（约 37ms/query）对于所有人机交互场景都绰绰有余，而且 BQ 根本不需要等待建图。

## 五、BQ 精查阀门选型建议

| 应用场景 | 推荐阀门 | 理由 |
| -------- | -------- | ---- |
| AI Agent 长期记忆 / 精准 RAG 知识库 | **5%** | 99.5% 接近无损召回，不容错失关键记忆 |
| 个人知识库语义搜索 | **5%** | 准确率优先，延迟不敏感 |
| 游戏 NPC 海量联想 / 模糊推荐 | **1%** | 89% 召回率已够用，22 倍加速换极致流畅 |
| 高维大规模（>10 万）实时检索 | **1%** | 大池子稀释了 1% 误杀率，性价比最高 |

## 六、总结

1. **BQ 精查 5% 是黄金参数**：在 5 万 ~ 50 万各规模下均保持 97% ~ 99.5% 的高召回率，速度较暴力提升 3 ~ 12 倍。
2. **BQ 精查 1% 是激进超车挡**：20 万规模可达 22.4 倍加速、174 QPS，召回率依然有 89.3%，大规模高吞吐场景极具竞争力。
3. **50 万规模建库速度碾压 HNSW 约 500 倍**，内存仅需一半，冷启动毫秒级。
4. HNSW 的搜索速度优势（3.3x QPS）以建库时间（×500）和内存代价（×2）为代价，在嵌入式 AI 领域这笔账怎么算都亏。

> _"用 1.69 小时的建图时间，换取 3.3 倍的查询提速——这笔账，在嵌入式 AI 领域，怎么算都亏。"_  
> —— TriviumDB 设计档案

## 七、v0.6.0 全 API 查询速度基准测试

> 测试环境：Windows 11 x86_64 / Intel i5-14400F (6P+4E) / Rust 1.93.1 (release, LTO=true)  
> 测试数据：5,000~10,000 节点 / 128 维 / 每节点 5 字段 JSON + 平均 3 条出边  
> 工具：Criterion v0.5（统计采样，排除 I/O 噪声）  
> 运行命令：`cargo bench --bench bench_queries`

### 7.1 CRUD 快速路径

| 操作 | 延迟 | 等效 QPS | 评价 |
|------|------|---------|------|
| `contains(id)` | **16 ns** | 62,500,000 | ⚡ HashMap O(1)，接近 L1 cache hit 的理论下限 |
| `get_edges(id)` | **153 ns** | 6,535,947 | ⚡ 邻接表直查，几乎零开销 |
| `get_payload(id)` | **369 ns** | 2,710,027 | ✅ 含 JSON Value 拷贝，依然亚微秒 |

> 📊 **对比**：Redis 单次 GET 约 50~100 µs（网络 RTT），TriviumDB 的 `contains` 比 Redis 快 **3000~6000 倍**（进程内 vs 网络，不完全可比，但说明嵌入式优势巨大）。SQLite `SELECT ... WHERE rowid=?` 约 1~5 µs，TriviumDB `get_payload` 比 SQLite 主键查询快 **3~14 倍**。

### 7.2 TQL 只读查询

| 操作 | 延迟 | 等效 QPS | 评价 |
|------|------|---------|------|
| `MATCH (a {id:N})-[]->(b) WHERE b.x>5 RETURN b` | **3.8 µs** | 263,158 | ⚡ ID 锚定 + WHERE 过滤，微秒级 |
| `MATCH (a {id:N})-[]->(b) RETURN b` | **7.1 µs** | 140,845 | ⚡ 单跳图遍历，含 TQL 解析开销 |
| `FIND {type:"event", active:true} RETURN *` (5K) | **780 µs** | 1,282 | ✅ 复合条件过滤，布隆拦截器生效 |
| `FIND {type:"event"} RETURN *` (5K) | **1.17 ms** | 855 | ✅ 全表扫描 + Bit-Tag 加速 |
| `FIND {type:"event"} RETURN *` (10K，无索引) | **2.97 ms** | 337 | 线性扩展，符合预期 |
| `FIND {type:"event"} RETURN *` (10K，有索引) | **3.05 ms** | 328 | ⚠️ 索引未生效（见下方分析） |
| `MATCH (a {type:"event"})-[]->(b) RETURN b` | **11.2 ms** | 89 | 🐢 无 ID 锚定的全表扫描，正常 |

> 📊 **对比**：
> - **MongoDB** 的 `find({type:"event"})` 在 5K 文档上约 **0.5~2 ms**（取决于是否有索引），TriviumDB FIND 约 1.17 ms，在同一数量级。
> - **Neo4j** 的 `MATCH (a)-[]->(b) RETURN b` 在 5K 节点图上约 **5~50 ms**（取决于连接数），TriviumDB 的 7.1 µs ID 锚定遍历快 **700~7000 倍**。
> - 关键优势是 **ID 锚定短路**：只要起点 ID 已知，TQL MATCH 就是 O(1) 定位 + O(degree) 扩散，与数据总量无关。

### 7.3 TQL DML 写操作

| 操作 | 延迟 | 等效 QPS | 评价 |
|------|------|---------|------|
| `CREATE (a {name:"x", type:"y"})` | **9.6 µs** | 104,167 | ⚡ 含 TQL 解析 + WAL 写入 |
| `MATCH (a {id:N}) SET a.val == X` | **10.6 µs** | 94,340 | ⚡ ID 定位 + Payload 更新 + WAL |
| `MATCH (a {type:"X"}) DETACH DELETE a` | **74 µs** | 13,514 | ✅ 批量删除 10 节点 + 边清理 |

> 📊 **对比**：SQLite 单次 INSERT 约 **20~100 µs**（WAL 模式），TriviumDB CREATE 约 9.6 µs，**快 2~10 倍**。

### 7.4 索引管理

| 操作 | 延迟 | 评价 |
|------|------|------|
| `create_index` + `drop_index` (5K 节点) | **607 µs** | ✅ 全量回填 5000 个 JSON 字段仅需 0.6ms |

### 7.5 向量搜索

| 操作 | 延迟 | 等效 QPS | 评价 |
|------|------|---------|------|
| `search(top_k=10)` 5K × 128d | **490 µs** | 2,041 | ✅ BQ 粗排 + 精排 |
| `search(top_k=10, expand=2)` 5K × 128d | **575 µs** | 1,739 | ✅ 向量检索 + 2 跳图扩散，仅多 17% 开销 |

> 📊 **换算吞吐**：5K × 128d 下的 2041 QPS 意味着 **~1020 万次向量比较/秒**。对于嵌入式场景（无 GPU、纯 CPU）这是优秀水平。作为参考，FAISS 的 IndexFlatIP 在类似规模下约 **3000~5000 QPS**（纯 SIMD 暴力搜索，无 WAL/持久化开销）。

### 7.6 速度分级总结

| 速度等级 | 延迟范围 | 包含操作 |
|----------|---------|---------|
| 🟢 **极速** | < 1 µs | `contains`, `get_edges`, `get_payload` |
| 🟢 **优秀** | 1~10 µs | TQL MATCH (ID 锚定), CREATE, SET |
| 🟡 **良好** | 10~100 µs | TQL DELETE, TQL MATCH+WHERE |
| 🟡 **正常** | 100 µs~1 ms | 向量搜索 5K, 索引回填, FIND (复合条件) |
| 🟠 **可接受** | 1~5 ms | FIND 全表扫描, 向量搜索 (大规模) |
| 🔴 **需优化** | > 10 ms | 无 ID 的属性全表 MATCH |

### 7.7 已知问题

**属性索引未加速 FIND 查询**：10K 节点上的 `FIND {type:"event"}` 有无索引延迟几乎相同（2.97ms vs 3.05ms）。原因是当前 TQL FIND 执行器在索引命中后仍需反查 Payload 构造返回结果，I/O 瓶颈在结果集组装而非候选集筛选。后续版本可通过覆盖索引（Covering Index）或延迟物化（Late Materialization）优化。

## 九、综合评价

对于一台 **i5-14400F** ：

1. **CRUD 快速路径达到 L1 cache 级速度**（16ns 的 `contains`），说明内存布局设计合理
2. **TQL 写操作 ~10 µs**，比多数嵌入式数据库的 WAL 写入更快
3. **向量搜索 2000 QPS @ 5K**，纯 CPU 下属于上游水平
4. **图遍历 ID 锚定 7 µs**，完胜 Neo4j 等外置图数据库的网络开销

> _"14400F 不是渣机，是性价比之王。而这些数字证明 TriviumDB 充分压榨了它的每一个时钟周期。"_

# TriviumDB v0.4.81 索引性能与精度评测报告

> 测试环境：Windows 11 x86_64 / Rust 1.93.1 (release profile, LTO=true)  
> 测试时间：2026-03-30  
> 测试数据：固定随机种子（`StdRng::seed_from_u64(42)`），向量已归一化

---

## 一、背景

TriviumDB v0.4.81 引入了全新的 **ERPC（Electoral Residual Projection Code）** 近似索引，
替代了原有的 HNSW 方案。本报告通过三组实验，量化评估 ERPC 在不同数据规模和维度下的
**精度（Recall@K）** 与 **性能（QPS / 延迟）** 表现，并与工业级 HNSW 实现（`instant-distance`）做正面对比。

---

## 二、ERPC vs BruteForce：精度与加速比

### 2.1 小规模（10,000 条 / 128 维）

| 策略               | 扫描比例 | Recall@10 | QPS   | 加速比    |
| ------------------ | -------- | --------- | ----- | --------- |
| BruteForce（基准） | 100%     | 100.00%   | 1,060 | 1.00x     |
| BQ 1%（极限）      | 1%       | 62.20%    | 3,088 | **2.91x** |
| BQ 5%（极速）      | 5%       | 91.00%    | 3,453 | **3.26x** |
| BQ 15%（均衡）     | 15%      | 98.90%    | 2,723 | **2.57x** |

> **Note**
> 低维度（128D）下向量密度高，BQ 签名区分度相对偏低，1% 极限模式下精度有所下降。
> 此场景建议使用 5%\~15% 配置。

### 2.2 大规模高维度（100,000 条 / 1536 维）

| 策略               | 扫描比例 | Recall@10   | QPS       | 加速比     |
| ------------------ | -------- | ----------- | --------- | ---------- |
| BruteForce（基准） | 100%     | 100.00%     | 12.2      | 1.00x      |
| BQ 1%（极限）      | 1%       | **86.00%**  | **159.5** | **13.11x** |
| BQ 5%（极速）      | 5%       | **98.00%**  | **111.1** | **9.13x**  |
| BQ 15%（均衡）     | 15%      | **100.00%** | **55.1**  | **4.53x**  |

> **关键结论**
> 高维度（1536D）正是 ERPC 架构的绝对“甜点区”！
>
> - 在仅扫描 **5%** 数据的情况下，实现了 **98% 的召回率** 和 **>9 倍的速度提升**。
> - 甚至在极其苛刻的 **1%** 扫描下，依然能拿到 **86%** 的精度。这表明二进制量化（BQ）签名在高维稀疏空间具有极强的特征捕捉能力。

---

## 三、ERPC vs HNSW：极限生存战

本测试对比 TriviumDB 原生的 ERPC 架构与业内知名 Rust HNSW 库（`instant-distance`）。
测试集：**20,000 条数据 / 1536 维 (OpenAI/Llama 标准)**。

| 模块 / 指标       | HNSW (instant-distance)    | ERPC (TriviumDB, 1% 扫描) | ERPC (TriviumDB, 5% 扫描) | 对比优势                       |
| ----------------- | -------------------------- | ------------------------- | ------------------------- | ------------------------------ |
| **构建时长**      | **256,160 ms** (约4.3分钟) | **530 ms**                | 530 ms                    | **ERPC 快了近 480 倍 🚀**      |
| **查询耗时**      | 11.23 ms /次               | **0.97 ms** /次           | 1.84 ms /次               | **ERPC (1%) 快了 11.5 倍 ⚡**  |
| **内存储存**      | 大量结构指针与高额内存开销 | **极小（线性序列）**      | 极小（同左）              | ERPC 的轻重级别完胜 🍃         |
| **持久化 & 更新** | O(N) 图重建成本高昂        | **增量无损实时写入 Mmap** | 无极扩容（Mmap）          | 天生适用 Agent 动态记忆片段 🐈 |

> **深入分析（Cache Miss效应）**
> 实验表明，在 **1536维** 这种高维空间中，HNSW 获取邻居指针所需的非连续性内存跳跃（Graph Traversal）导致了高昂的 Cache Miss。
> 而 ERPC 通过莫顿码（Morton Code）做连续的 Z-Order 定位后，配合连续内存上的粗筛/精算，极大地迎合了现代 CPU 的预取机制（Prefetching）。这使得主观上被认为 O(N\*ratio) 很慢的线性近似扫描，在吞吐量上反超了O(logN) 的 HNSW！

## 四、总结与工程美学：Hill 曲线的胜利

通过对 1 万、5 万、10 万级别的主成分分析与回归，TriviumDB 中的 ERPC 算法核心调用了 **Hill 方程 (代数 Sigmoid)** 进行超参数聚类自适应调节：
[f(N) = Max * (N^p) / (Half^p + N^p)]，配合 `p=1.2` 这一绝妙的魔术数字。

- **动态升降档**：在早期（N < 90,000）保持算力克制，维持写入的超低延迟；在后期（N > 200k）激增扫描比例控制精度损失。
- **高维压制力**：向量维数越高（如 1536D），传统图算法的内存跨步越成为致命弱点。而在连续 Mmap 文件流上的 Z-Order 一维化让 TriviumDB 跨越了这个维度的陷阱！
- **重器新锋**：ERPC 是天生适配“随带走（Portability）”、“小内存（Low-RAM）”、“秒拉取（Instant Load）”的设计典范，也是作为 **大模型 AI 实体认知/Agent 长线私有记忆池** 的绝佳底层引擎。
